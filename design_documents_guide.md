# 要件定義・基本設計・詳細設計・仕様書の整理

## 1. この資料の目的

この資料は、開発プロジェクトにおける以下のドキュメントの役割と運用方法を整理するためのものです。

- 要件定義書
- 基本設計書
- 詳細設計書
- 仕様書
- 差分設計書
- リリースノート
- ADR

特に、以下の混乱を防ぐことを目的とします。

- 基本設計と詳細設計の違いが分からない
- 詳細設計書と仕様書の違いが分からない
- 新機能追加時に既存資料を更新するべきか分からない
- 差分資料と大本資料の関係が分からない
- 設計書が古くなり、信用できない資料になる

---

## 2. 全体の考え方

設計書には大きく2種類あります。

```text
1. 現在地を示す資料
2. 変更過程を示す資料
```

### 2.1 現在地を示す資料

現在のシステムがどうなっているかを表す資料です。

外から入ってきた人、保守担当、PM、開発者が最初に読むべき資料です。

代表例：

- 要件定義書
- 基本設計書
- 詳細設計書
- 仕様書
- API仕様書
- DB定義書
- マニュアル

これらは、最終的に必ず最新状態に保つ必要があります。

### 2.2 変更過程を示す資料

今回の開発で何を変えるのか、なぜ変えるのかを示す資料です。

代表例：

- 差分設計書
- 変更設計書
- リリースノート
- ADR

これらは、開発中のレビューや変更履歴の保存に使います。

---

## 3. 各ドキュメントの役割

| 資料 | 役割 | 主な読者 |
|---|---|---|
| 要件定義書 | なぜ作るのか、何を満たすべきかを書く | 顧客、PM、PL、開発者 |
| 基本設計書 | 何を作るか、外からどう見えるかを書く | 顧客、PM、PL、開発者 |
| 詳細設計書 | どう実装するかを書く | 開発者、レビュアー |
| 仕様書 | 現在その機能がどう動くかを書く | 顧客、QA、開発者、保守担当 |
| API仕様書 | APIのリクエスト・レスポンスを書く | フロントエンド、バックエンド、外部連携担当 |
| DB定義書 | テーブル・カラム・制約を書く | バックエンド、DB担当 |
| マニュアル | 利用者向けの操作説明を書く | 利用者、運用担当 |
| 差分設計書 | 今回の変更内容を書く | PM、PL、開発者、レビュアー |
| リリースノート | いつ何が変わったかを書く | 顧客、運用担当、開発者 |
| ADR | なぜその技術判断をしたかを書く | 開発者、アーキテクト、未来の保守担当 |

---

## 4. 要件定義書・基本設計書・詳細設計書・仕様書の違い

## 4.1 要件定義書

要件定義書は、システムが満たすべき要求を書く資料です。

主に以下を書きます。

- システムの目的
- 解決したい課題
- 利用者
- 業務要件
- 機能要件
- 非機能要件
- 制約条件
- 対象範囲
- 対象外範囲

要件定義書の中心は、**なぜ作るのか・何を満たすべきか**です。

---

## 4.2 基本設計書

基本設計書は、要件を満たすために、システムを外から見てどう作るかを書く資料です。

主に以下を書きます。

- システム概要
- 機能一覧
- 画面一覧
- 画面遷移
- 画面仕様の概要
- APIの概要
- 外部連携の概要
- 権限設計
- データ設計の概要
- バッチ概要
- メール・通知概要
- 非機能要件への対応方針

基本設計書の中心は、**何を作るか・外からどう見えるか**です。

### 基本設計に書きすぎない方がよい内容

以下は詳細設計寄りです。

- Controller名
- Service名
- Repository名
- 細かいSQL
- メソッド単位の処理
- 実装コードに近い疑似コード
- フレームワーク固有の細かい実装

---

## 4.3 詳細設計書

詳細設計書は、基本設計で決まった内容を、実装できる粒度に落とし込む資料です。

主に以下を書きます。

- API詳細
- リクエスト・レスポンス
- 入力チェック
- 業務ルール
- 処理フロー
- 例外処理
- トランザクション範囲
- 使用テーブル
- CRUD表
- SQL・クエリ仕様
- クラス・モジュール構成
- ログ出力
- 外部連携の詳細
- リトライ方針
- テスト観点

詳細設計書の中心は、**どう実装するか**です。

---

## 4.4 仕様書

仕様書は、現在その機能がどう動くかを説明する資料です。

主に以下を書きます。

- 機能概要
- 操作方法
- 入力項目
- 表示項目
- 出力項目
- 正常時の振る舞い
- 異常時の振る舞い
- 権限ごとの操作可否
- 状態ごとの操作可否
- エラーメッセージ
- CSVやPDFなどの出力仕様

仕様書の中心は、**利用者・外部システムから見てどう振る舞うか**です。

---

## 5. 仕様書と詳細設計書の違い

最も混乱しやすいのが、仕様書と詳細設計書の違いです。

結論は以下です。

```text
仕様書：
  その機能が外から見てどう振る舞うか

詳細設計書：
  その機能を中でどう実装するか
```

もう少し具体的に言うと、以下のようになります。

| 観点 | 仕様書 | 詳細設計書 |
|---|---|---|
| 目的 | 何ができるかを説明する | どう作るかを説明する |
| 視点 | 利用者・外部システム視点 | 開発者・内部実装視点 |
| 主な読者 | 顧客、PM、QA、開発者、保守担当 | 開発者、レビュアー |
| 内容 | 画面、項目、操作、制約、エラー | API、DB、SQL、クラス、分岐、例外、ログ |
| コード寄りか | 低い | 高い |
| 更新条件 | 外から見た振る舞いが変わる時 | 内部処理や実装方式が変わる時 |

---

## 6. 仕様書に書くこと

仕様書には、利用者や外部システムから見える振る舞いを書きます。

例：ユーザー登録機能

```md
# ユーザー登録仕様

## 概要
管理者は新しいユーザーを登録できる。

## 入力項目

| 項目 | 必須 | 内容 |
|---|---|---|
| 名前 | 必須 | 255文字以内 |
| メールアドレス | 必須 | メール形式 |
| 権限 | 必須 | 管理者 / 一般ユーザー |

## 正常時

- 正しい内容を入力して登録すると、ユーザーが作成される
- 登録完了メッセージを表示する
- ユーザー一覧に登録したユーザーが表示される

## 異常時

- 名前が未入力の場合、エラーを表示する
- メールアドレス形式が不正な場合、エラーを表示する
- 既に使われているメールアドレスは登録できない

## 権限

- 管理者のみ登録できる
- 一般ユーザーは登録できない
```

仕様書は、**何ができるか・何ができないか**を中心に書きます。

---

## 7. 詳細設計書に書くこと

詳細設計書には、実装するために必要な情報を書きます。

例：ユーザー登録機能

```md
# ユーザー登録 詳細設計

## API

POST /api/admin/users

## 認証・認可

- auth middlewareでログイン確認
- policyでadmin権限を確認
- 権限がない場合は403を返す

## 入力チェック

- name: required|string|max:255
- email: required|email|max:255|unique:users,email
- role: required|in:admin,user

## 処理フロー

1. リクエストを受け取る
2. 認証を確認する
3. 管理者権限を確認する
4. 入力値を検証する
5. usersテーブルに登録する
6. operation_logsテーブルに操作ログを登録する
7. 登録結果をJSONで返す

## トランザクション

- users登録
- operation_logs登録
- どちらかに失敗した場合はロールバックする

## 使用テーブル

- users
- operation_logs

## レスポンス

HTTP Status: 201 Created

```json
{
  "id": 1,
  "name": "山田太郎",
  "email": "yamada@example.com",
  "role": "user"
}
```
```

詳細設計書は、**実装するために必要な情報**を中心に書きます。

## 8. 仕様書・API仕様書・詳細設計書の違い

APIがある場合、仕様書・API仕様書・詳細設計書も混ざりやすいです。

| 資料 | 書く内容 |
|---|---|
| 仕様書 | その機能がどう動くか |
| API仕様書 | APIのリクエスト・レスポンス |
| 詳細設計書 | APIの中でどう処理するか |

例：ユーザーCSV出力機能

### 仕様書

```text
管理者はユーザー一覧をCSV形式で出力できる。
現在の検索条件に一致するユーザーが出力対象になる。
```

### API仕様書

```text
GET /api/admin/users/export

Query Parameters:
- keyword
- role
- created_from
- created_to

Response:
- Content-Type: text/csv
```

### 詳細設計書

```text
1. admin権限を確認する
2. query parameterを検証する
3. usersテーブルを検索する
4. CSVを生成する
5. operation_logsに出力履歴を保存する
6. CSVをレスポンスとして返す
```

## 9. 正常系・異常系には何を書くか

基本設計や仕様書に出てくる「正常系」「異常系」は、ユーザーや外部システムから見た振る舞いを書く場所です。

### 9.1 正常系に書くこと

正常系には、ユーザーが正しく操作した場合に、システムがどう振る舞うかを書きます。

書く内容：

- どんな操作をしたら
- どんな条件なら成功して
- 何が登録・更新・表示・削除されて
- 画面やAPIが何を返すか
- 次にどこへ遷移するか
- ユーザーに何が見えるか

例：ユーザー登録機能

| No | 操作・条件 | システムの動作 | 結果 |
|---|---|---|---|
| 1 | 必須項目をすべて入力して登録ボタンを押す | 入力内容を検証し、ユーザー情報を登録する | 登録完了メッセージを表示する |
| 2 | 登録完了後 | ユーザー一覧画面へ遷移する | 登録したユーザーが一覧に表示される |
| 3 | メール通知が有効な場合 | 登録完了メールを送信する | ユーザーにメールが届く |

---

### 9.2 異常系に書くこと

異常系には、想定される失敗パターンと、そのときのシステムの振る舞いを書きます。

書く内容：

- 入力が不正
- 権限がない
- データが存在しない
- 重複している
- 状態的に操作できない
- 外部APIが失敗した
- DB登録に失敗した

例：ユーザー登録機能

| No | 条件 | システムの動作 | 表示・返却内容 |
|---|---|---|---|
| 1 | 必須項目が未入力 | 登録処理を行わない | 「名前を入力してください」を表示 |
| 2 | メールアドレス形式が不正 | 登録処理を行わない | 「メールアドレスの形式が正しくありません」を表示 |
| 3 | 既に登録済みのメールアドレス | 登録処理を行わない | 「このメールアドレスは既に使用されています」を表示 |
| 4 | 権限がないユーザーが登録しようとする | 処理を中断する | 403 / 権限がありません |
| 5 | DB登録に失敗する | 登録処理をロールバックする | 「登録に失敗しました」を表示 |

---

## 10. 基本設計と詳細設計における正常系・異常系の違い

| 書く場所 | 書く内容 |
|---|---|
| 基本設計の正常系・異常系 | ユーザー視点・業務視点での成功/失敗パターン |
| 詳細設計の処理詳細 | Controller、UseCase、Repository、DB更新、トランザクションなど |
| API仕様書 | HTTP status、request、response、error code |
| テスト仕様書 | 入力値、操作手順、期待結果 |

### 基本設計で書く例

```text
正しい入力内容で登録を実行した場合、ユーザー情報を登録し、登録完了メッセージを表示する。
入力内容に不備がある場合、登録処理は行わず、該当項目にエラーメッセージを表示する。
```

### 詳細設計で書く例

```text
ControllerでRequestを受け取り、UseCaseを呼び出す。
UseCase内で認可チェック、入力チェック、DB登録、操作ログ登録を行う。
DB登録または操作ログ登録に失敗した場合はロールバックする。
```

## 11. 新機能追加時の資料更新方針

新機能を追加する場合、最終的には以下の大本資料を最新化します。

- 要件定義書
- 基本設計書
- 詳細設計書
- 仕様書
- API仕様書
- DB定義書
- マニュアル

ただし、開発中にいきなり大本資料を直接編集するだけだと、今回何が変わったのか分かりづらくなります。

そのため、実務では以下の2段階に分けるのが安全です。

```text
1. 開発中は差分設計書を作る
2. リリース前後に大本資料へ反映する
```

---

## 12. 新機能追加時の流れ

```text
開発中：
  変更差分の資料を作る
  ↓
  その差分をレビューする
  ↓
  実装する

リリース前後：
  差分内容を大本の資料へ反映する
  ↓
  大本の要件定義・基本設計・詳細設計・仕様書を最新化する
  ↓
  差分資料は変更履歴として残す
```

重要なのは、以下の考え方です。

```text
外から入ってきた人が読む資料：
  大本の最新版資料

開発中にレビューする資料：
  今回の差分資料
```

## 13. 差分設計書に書く内容

新機能追加時は、以下のような差分設計書を作ると運用しやすいです。

例：ユーザーCSVエクスポート機能追加

```md
# ユーザーCSVエクスポート機能 追加設計

## 1. 背景・目的
管理者がユーザー一覧をCSVで出力できるようにする。

## 2. 要件

- 管理者のみCSV出力できる
- 出力項目はID、名前、メールアドレス、登録日時
- 削除済みユーザーは出力しない

## 3. 基本設計差分

- ユーザー一覧画面に「CSV出力」ボタンを追加
- 管理者権限のみ表示
- 出力形式はCSV

## 4. 詳細設計

- API: GET /api/admin/users/export
- 認証: 必須
- 認可: adminのみ
- usersテーブルから対象データを取得
- CSVをstreamDownloadで返却

## 5. 影響範囲

- ユーザー一覧画面
- ユーザー管理API
- 権限設計
- ファイル出力仕様
- 管理者マニュアル

## 6. テスト観点

- 管理者はCSV出力できる
- 一般ユーザーはCSV出力できない
- 削除済みユーザーは含まれない
```

この差分資料をレビューして、問題なければ実装します。

その後、確定した内容を大本資料へ反映します。

---

## 14. 大本資料への反映例

差分資料で決まった内容は、最終的に大本資料へ反映します。

### 要件定義書への反映例

```md
## 機能要件

- 管理者はユーザー一覧をCSV形式で出力できること。
```

### 基本設計書への反映例

```md
## 機能一覧

| 機能ID | 機能名 | 概要 |
|---|---|---|
| F-023 | ユーザーCSV出力 | 管理者がユーザー一覧をCSVで出力する |
```

```md
## 画面仕様：ユーザー一覧

- CSV出力ボタンを表示する
- 管理者権限の場合のみ表示する
```

### 詳細設計書への反映例

```md
## ユーザーCSV出力 API

- GET /api/admin/users/export
- admin権限のみ実行可能
- usersテーブルから削除済み以外のユーザーを取得
- CSV形式で返却する
```

## 15. 機能追加・変更時の更新対象

| 変更内容 | 要件定義 | 基本設計 | 詳細設計 | 仕様書 | マニュアル |
|---|---:|---:|---:|---:|---:|
| 新機能追加 | ○ | ○ | ○ | ○ | ○ |
| 機能廃止 | ○ | ○ | ○ | ○ | ○ |
| 新画面追加 | △ | ○ | ○ | ○ | ○ |
| 画面項目追加 | △ | ○ | ○ | ○ | ○ |
| API追加 | △ | ○ | ○ | ○ | △ |
| DBカラム追加 | △ | △ | ○ | ○ | × |
| バリデーション追加 | △ | △ | ○ | ○ | △ |
| 権限変更 | ○ | ○ | ○ | ○ | ○ |
| エラーメッセージ変更 | × | △ | ○ | ○ | △ |
| 内部リファクタリング | × | × | △ | × | × |
| ライブラリ変更 | × | × | △ | × | × |
| インフラ変更 | △ | ○ | ○ | △ | × |

凡例：

- ○：基本的に更新する
- △：影響があれば更新する
- ×：原則更新しない

---

## 16. PR完了条件に入れるべきもの

設計書を腐らせないためには、PRやリリースの完了条件にドキュメント更新を含めるべきです。

例：

```text
PRの完了条件：
  実装が終わっている
  テストが通っている
  差分設計がある
  大本資料に反映されている
  リリースノートが更新されている
```

これをやらないと、以下のようになります。

```text
差分設計はある
でも大本資料に反映されていない
↓
半年後、誰も正しい仕様が分からない
↓
コードを読むしかない
↓
設計書が信用されなくなる
```

## 17. おすすめのディレクトリ構成

個人開発から中小規模案件であれば、巨大な1枚の設計書より、Markdownで分割管理する方が運用しやすいです。

```text
docs/
  current/
    requirements.md
    basic-design.md
    detail-design.md
    specs/
      screens/
      api/
      db/
      permissions/
      errors/

  changes/
    2026-06-22-add-user-export.md
    2026-07-10-remove-legacy-login.md

  releases/
    v1.1.0.md
    v1.2.0.md

  adr/
    0001-use-refresh-token-cookie.md
    0002-use-outbox-pattern.md

  manuals/
    admin-manual.md
    user-manual.md
```

### 各ディレクトリの役割

| ディレクトリ | 役割 |
|---|---|
| current/ | 現在の最新版資料 |
| current/specs/ | 現在の仕様書群 |
| changes/ | 変更差分・追加設計 |
| releases/ | リリースごとの変更履歴 |
| adr/ | 技術判断の理由 |

## 18. 機能単位で分ける場合の構成

機能単位で資料をまとめたい場合は、以下の構成も使えます。

```text
docs/features/user-management/
  00-overview.md        # 基本設計寄り
  01-screen-spec.md     # 画面仕様
  02-api-spec.md        # API仕様
  03-detail-design.md   # 詳細設計
  04-test-cases.md      # テスト観点
  05-manual.md          # 操作説明
```

この形は、新機能追加や機能廃止時に影響範囲が見えやすいです。

## 19. 判断基準

仕様書と詳細設計書のどちらに書くべきか迷ったら、以下で判断します。

### 19.1 仕様書に書くべきもの

以下の質問にYESなら、仕様書に書きます。

```text
それは利用者や外部システムが知るべきことか？
```

例：

- 入力できる文字数
- ボタンを押した後の結果
- エラーメッセージ
- 権限ごとにできる操作
- CSVの出力項目
- APIのレスポンス形式
- 画面に表示される項目

---

### 19.2 詳細設計書に書くべきもの

以下の質問にYESなら、詳細設計書に書きます。

```text
それは実装者がコードを書くために知るべきことか？
```

例：

- どのテーブルを読むか
- どの順番で処理するか
- どこでトランザクションを張るか
- どのRepositoryを使うか
- 失敗時にどのログを出すか
- 外部API失敗時にリトライするか
- キューに積むか同期処理するか
- 例外発生時にロールバックするか

## 20. 実務上の注意点

## 20.1 詳細設計書をコードの日本語訳にしない

詳細設計書は、コードをそのまま日本語にした資料ではありません。

悪い例：

```text
UserControllerのstoreメソッドでUserServiceのcreateメソッドを呼び出し、
UserRepositoryのinsertメソッドでusersテーブルにINSERTする。
```

これが必要な場合もありますが、書きすぎると実装変更のたびに資料が壊れます。

詳細設計で本当に重要なのは以下です。

- どの条件で処理を止めるのか
- どのデータを読む・書くのか
- どこからどこまでがトランザクションなのか
- 外部連携が失敗したらどうするのか
- 権限・状態・期限などの業務ルールは何か
- テストで何を確認すべきか

---

## 20.2 大本資料だけ直接更新しない

新機能追加時に、いきなり大本の基本設計書や詳細設計書だけを編集すると、今回何が変わったのか分かりにくくなります。

そのため、まずは差分設計書を作るべきです。

```text
差分設計書 = 今回何を変えるか
大本資料 = 現在のシステムがどうなっているか
```

この2つを分けることで、レビューもしやすくなります。

---

## 20.3 差分資料だけで終わらせない

差分資料だけ作って、大本資料を更新しないのも危険です。

外から入ってきた人が読むのは、基本的に大本の資料です。

そのため、最終的には以下を最新にします。

- 要件定義書
- 基本設計書
- 詳細設計書
- 仕様書
- API仕様書
- DB定義書
- マニュアル

---

## 21. 最終まとめ

要点は以下です。

```text
要件定義書：
  なぜ作るか、何を満たすか

基本設計書：
  何を作るか、外からどう見えるか

詳細設計書：
  どう実装するか

仕様書：
  現在その機能がどう動くか

差分設計書：
  今回何を変えるか

リリースノート：
  いつ何が変わったか

ADR：
  なぜその技術判断をしたか
```

新機能追加時の正しい運用は以下です。

```text
1. 差分設計書を作る
2. 差分設計をレビューする
3. 実装する
4. テストする
5. 大本の要件定義・基本設計・詳細設計・仕様書を更新する
6. リリースノートを書く
7. 差分設計書は履歴として残す
```

外から入ってきた人が読むのは、常に大本の最新版資料です。

開発中に読むのは、今回の差分資料です。

この2つを分けて運用すれば、設計書が腐りにくくなります。

## 22. 一言でまとめる

```text
仕様書は「外から見た振る舞い」。
詳細設計書は「中での作り方」。

開発中は差分設計で進める。
リリース前後に大本資料へ反映する。

差分資料は変更履歴。
大本資料は現在地。
```
